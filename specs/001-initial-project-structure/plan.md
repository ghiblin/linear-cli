# Implementation Plan: Initial Project Structure

**Branch**: `001-initial-project-structure` | **Date**: 2026-04-21 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `specs/001-initial-project-structure/spec.md`

## Summary

Establish a compilable, DDD-layered Rust skeleton for the Linear CLI. The skeleton wires up the four-layer module hierarchy (domain → application → infrastructure → cli), configures all quality gates (fmt, clippy, tests), sets up the CI pipeline for four target platforms, and exposes stub implementations of `linear --version`, `linear --help`, `linear issue list`, and `linear team list`. No real Linear API connection is included; infrastructure stubs return placeholder responses sufficient to demonstrate the architectural boundary.

## Technical Context

**Language/Version**: Rust 1.85.0 (stable, pinned via `rust-toolchain.toml`)  
**Primary Dependencies**: tokio (async runtime), reqwest+rustls (HTTP client), cynic (GraphQL, build-time schema validation), clap v4 derive API (CLI), serde+serde_json (serialization), thiserror (library error types), anyhow (binary entry point only), keyring (system keychain), tracing+tracing-subscriber (logging), mockall (trait mocking in tests), insta (snapshot tests)  
**Storage**: N/A — no persistent storage; API key stored in system keychain via `keyring`  
**Testing**: `cargo test`; unit tests co-located in `src/` with `#[cfg(test)]`; integration tests under `tests/integration/`; e2e tests under `tests/e2e/` (opt-in, require `LINEAR_TEST_API_KEY`)  
**Target Platform**: macOS arm64, macOS x86_64, Linux x86_64, Linux arm64  
**Project Type**: CLI binary  
**Performance Goals**: Release binary under 20 MB; local quality-check suite completes under 2 minutes  
**Constraints**: No `unsafe`; no `unwrap()`/`expect()` in production paths; `clippy -D warnings`; no blocking calls in async contexts  
**Scale/Scope**: Single binary crate; initial skeleton; domain entities are stubs

**Structure Decision**: Single binary crate with module-based DDD layering. Rust's `pub(crate)` / `pub(super)` visibility enforces one-way dependencies without workspace overhead. Migration to a Cargo workspace is deferred until library extraction is needed (see research.md Decision 2).
